## Reading: Best Practices for Log Collection and Management

A **log** is a record of events within an organization's systems, originally used mainly for troubleshooting but now central to security investigations via **log analysis** — the process of examining logs to identify events of interest, supporting the 5 W's (who, what, when, where, why) of an investigation.

### Types of Logs

Network, system, application, security, and authentication logs — generated by the corresponding devices/software (firewalls/routers, OS, software apps, antivirus/IDS, and login events, respectively).

### Log Details

Logs typically include a date, time, location, action, and author. **Verbose logging** records additional detail beyond the default.

### Log Management

The process of collecting, storing, analyzing, and disposing of log data.

- **What to log:** Choose sources most likely to contain useful information for the event of interest; some data (PII) requires special handling and, depending on jurisdiction, may not be loggable at all.
- **The issue with overlogging:** Logging everything is tempting but increases storage/maintenance costs and system load, and makes it harder to find important events.
- **Log retention:** Some regulated industries (public sector/FISMA, healthcare/HIPAA, financial services/PCI DSS, GLBA, SOX) require specific log retention periods.
- **Log protection:** Attackers sometimes modify logs to mislead investigators or hide activity. Storing logs on a **centralized log server**, rather than locally, adds a barrier between an attacker and the log data, helping maintain log integrity.

**Key takeaway:** Deciding what to log, how long to retain it, and how to protect it are all deliberate management choices — logging everything indiscriminately can actually reduce a SIEM's usefulness.

## Reading: Overview of Log File Formats

### JSON (JavaScript Object Notation)

A lightweight format for storing/transmitting data, common in web technologies and cloud environments. Uses **key-value pairs** (`"key": "value"`), commas to separate data, double quotes around string values, **curly brackets `{}`** to enclose an **object** (a comma-separated list of key-value pairs), and **square brackets `[]`** to enclose an **array** (a comma-separated ordered list).

### Syslog

Refers to three related things: the **protocol** for transporting logs to a centralized server (port 514 plaintext, 6514 encrypted), the **service** that consolidates/forwards logs, and the **log format** itself — the native format on Unix systems, consisting of a **header** (timestamp, hostname, application name, message ID), **structured-data** (additional key-value info in brackets), and a **message**. A **priority (PRI)** field in angle brackets indicates urgency (lower = more urgent). Syslog headers can be combined with JSON/XML formats or used with custom formats.

### XML (eXtensible Markup Language)

The native format on Windows systems, using paired **tags** (`<tag>...</tag>`), **elements** (a tag plus its enclosed data, always starting with one root element containing child elements), and **attributes** (extra info inside a tag, always quoted, e.g., `<Data Name='SubjectUserSid'>...`).

### CSV (Comma Separated Value)

Uses commas to separate data values; a value's position corresponds to its field name, though field names themselves may not be included in the log — understanding what fields the source device includes is critical to reading a CSV log correctly.

### CEF (Common Event Format)

Uses key-value pairs, structured as: `CEF:Version|Device Vendor|Device Product|Device Version|Signature ID|Name|Severity|Extension`, with fields separated by pipes (`|`) and the Extension field written in key-value format. Often transported via Syslog, which prepends a timestamp and hostname to the CEF message.

**Key takeaway:** There's no single standard log format — analysts need to recognize and interpret JSON, Syslog, XML, CSV, and CEF, since logs from different sources will typically arrive in different formats.

## Reading: Detection Tools and Techniques

### Host-Based Intrusion Detection System (HIDS)

An application (agent) installed on a host/endpoint that monitors that specific device's internal activity — inbound/outbound traffic, file systems, resource usage, user activity — and alerts on anomalies like unauthorized software installation.

### Network-Based Intrusion Detection System (NIDS)

Installed at specific points in the network to inspect traffic from multiple devices, alerting on detected malicious traffic. Combining HIDS and NIDS gives a multi-layered, more comprehensive view of environment activity.

### Detection Techniques

- **Signature-based (signature analysis):** Compares activity against known **signatures** — patterns (byte sequences, IP addresses, etc.) associated with malicious activity, informed by IoCs from the Pyramid of Pain.
  - *Advantages:* Low false-positive rate, since it's matching known patterns.
  - *Disadvantages:* Signatures can be evaded via attacker modifications, require constant updates as new exploits emerge, and can't detect unknown threats like zero-days.
- **Anomaly-based analysis:** Establishes a baseline of normal behavior (training phase), then flags current activity that deviates from it (detection phase).
  - *Advantages:* Can detect new/evolving threats that don't match a known signature.
  - *Disadvantages:* High false-positive rate (any deviation gets flagged, even benign ones), and if an attacker is already present during the training phase, their behavior gets baked into the "normal" baseline.

**Key takeaway:** NIDS monitors an entire network while HIDS monitors individual endpoints, and signature-based vs. anomaly-based detection each trade off differently between catching known threats reliably and catching unknown ones at the cost of more false positives.

## Reading: Overview of Suricata

**Suricata** is an open-source IDS, IPS, and network analysis tool with three main modes: **IDS** (network- or host-based monitoring/alerting), **IPS** (detecting and blocking malicious activity, requiring additional configuration like enabling IPS mode), and **network security monitoring (NSM)** (producing/saving relevant logs for forensics, incident response, or testing signatures).

### Rules

Suricata's terms "rule" and "signature" are used interchangeably, made up of three components:

- **Action:** What to do on a match — `alert`, `pass`, `drop`, or `reject`.
- **Header:** Network traffic info — source/destination IP, ports, protocol, traffic direction.
- **Rule options:** Customization options, which have a specific order that affects the rule's meaning.

Suricata ships with pre-written rules, but organizations are strongly encouraged to write custom rules tailored to their own infrastructure, since no one-size-fits-all approach exists — custom rules also help minimize false positives. **Rule order** matters: Suricata processes rules in the default order pass, drop, reject, alert, which affects the final verdict when conflicting rules match the same packet.

### Configuration File

Before deployment, Suricata's settings must be configured via `suricata.yaml` (using the YAML format), which customizes how Suricata interacts with the rest of the environment.

### Log Files

- **`eve.json`:** The standard, detailed log file (JSON format) — includes metadata like `flow_id` to correlate related events to a single network flow. Better suited for detailed analysis and SIEM ingestion.
- **`fast.log`:** A legacy format recording minimal alert info (basic IP/port details) — not well-suited for incident response or threat hunting.

**Key takeaway:** Suricata's power comes from customization — pre-written rules are a starting point, but writing/tuning signatures for a specific environment (and choosing the right log file for the task) is what makes it genuinely useful.

## Reading: Log Sources and Log Ingestion

Recapping the SIEM process — collect/aggregate, normalize, analyze — this reading focuses specifically on the first step.

### Log Ingestion

**Log ingestion** is the process of collecting/importing data from log sources into a SIEM. The SIEM creates and retains its own copy of the event data (rather than modifying the original source), providing a centralized platform for analysis.

### Log Forwarders

**Log forwarders** are software that automate collecting and sending log data — far more efficient than manually uploading data from potentially thousands of network devices. Some operating systems include native log forwarders; otherwise, third-party forwarding software can be installed and configured to specify which logs to forward and where (e.g., to a SIEM for processing/normalization). Many SIEM tools use proprietary log forwarders but can also integrate with open-source ones; the right choice depends on system requirements, compatibility, and existing infrastructure.

**Key takeaway:** Understanding where log sources come from — and how they get collected via forwarders — helps an analyst trace a security incident back to its source data.

## Reading: Search Methods with SIEM Tools

Different SIEM tools use different search methods; analysts should be ready to adapt across tools.

### Splunk Searches

Splunk uses **Search Processing Language (SPL)** to search/retrieve events from **indexes** (stored, processed event data) via the Search & Reporting app.

- **Basic search:** `index=main fail` retrieves events from the `main` index containing the term "fail."
- **Pipes (`|`):** Chain commands together, feeding one command's output as the next command's input — e.g., `index=main fail | chart count by host` transforms results into a chart of failure counts by host.
- **Wildcards (`*`):** Match variable characters — e.g., `fail*` matches "failed" or "failure" as well as "fail." Double quotes (`"login failure"`) search for an exact phrase.

### Chronicle Searches

- **Unified Data Model (UDM) Search:** The default search type, querying ingested/parsed/normalized data — faster since it searches indexed, structured UDM data. All UDM events include common fields: **entities** (nouns — device, user, or process context; every event has at least one), **event metadata** (basic event description, e.g., `metadata.event_type = "USER_LOGIN"`), **network metadata** (network-related event/protocol details), and **security results** (the security-related outcome of an event, e.g., "virus detected and quarantined").
- **Raw Log Search:** Searches unparsed raw logs directly (by username, filename, hash, etc.) when the normalized UDM data doesn't have what's needed — slower, but supports regular expressions for narrowing results.

**Key takeaway:** SPL and UDM/Raw Log searches are different tools for the same underlying goal — quickly and efficiently finding the specific event data needed to detect threats and respond to incidents.

## Personal Reflection

Walking through JSON, Syslog, XML, CSV, and CEF back to back made it click that "reading logs" isn't one skill — it's really several related skills, since the same event (like a failed login) could show up in any of these formats depending on the source device. The signature-based vs. anomaly-based detection tradeoff also reframed something I'd been treating as a solved problem: there's no single "best" detection method, just different tradeoffs between missing unknown threats and drowning in false positives — which is presumably why HIDS/NIDS and both detection techniques are so often layered together rather than used alone.
