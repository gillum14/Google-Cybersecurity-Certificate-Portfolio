# Course 1, Module 3 – Controls, Frameworks, Compliance, and Ethics

## Overview

This module covered how security frameworks, controls, and compliance regulations work together to manage risk, and introduced the ethical concepts and legal boundaries that guide how security professionals are permitted to respond to attacks.

## Reading: Controls, Frameworks, and Compliance

### How Controls, Frameworks, and Compliance Are Related

The CIA triad (confidentiality, integrity, availability) is the model that informs how organizations consider risk when setting up systems and policies. Security controls are safeguards designed to reduce specific security risks, and they're used alongside frameworks to ensure security goals are implemented correctly and regulatory compliance is met.

Security frameworks are guidelines for building plans to mitigate risk, built around four core components: identifying and documenting security goals, setting guidelines to achieve them, implementing strong security processes, and monitoring/communicating results. Compliance is the process of adhering to internal standards and external regulations.

### Specific Controls, Frameworks, and Compliance Standards

- **NIST:** A U.S. agency that develops voluntary compliance frameworks (e.g., NIST CSF, NIST RMF) that organizations worldwide use to manage risk.
- **FERC-NERC:** Applies to organizations working with electricity or the North American power grid; requires adherence to Critical Infrastructure Protection (CIP) Reliability Standards.
- **FedRAMP:** A U.S. federal program standardizing security assessment, authorization, and monitoring of cloud services.
- **CIS:** A nonprofit providing controls to safeguard systems and networks against attacks.
- **GDPR:** An E.U. regulation protecting the processing of E.U. residents' data; requires notifying affected individuals of a breach within 72 hours.
- **PCI DSS:** An international standard ensuring organizations that store/process credit card data do so securely, to reduce fraud.
- **HIPAA:** A 1996 U.S. federal law protecting patients' health information, governed by privacy, security, and breach notification rules. HITRUST is a related framework that helps institutions meet HIPAA compliance.
- **ISO:** Establishes international standards related to technology, manufacturing, and management across borders.
- **SOC 1 / SOC 2:** AICPA-developed reports assessing an organization's user access policies and financial/data compliance across roles (associate, supervisor, manager, executive, vendor, etc.).
- **Executive Order 14028:** A 2021 U.S. presidential order directing federal agencies and critical-infrastructure-adjacent third parties to improve cybersecurity.

**Key takeaway:** Frameworks, controls, and compliance work together to maintain a low level of organizational risk. Staying current on these — and researching others like the Gramm-Leach-Bliley Act and Sarbanes-Oxley Act — is an ongoing responsibility for security professionals.

## Reading: Ethical Concepts That Guide Cybersecurity Decisions

### U.S. Standpoint on Counterattacks

Deploying a counterattack on a threat actor is illegal in the U.S. under laws like the Computer Fraud and Abuse Act of 1986 and the Cybersecurity Information Sharing Act of 2015. Organizations may only defend — counterattacking is considered vigilantism and can escalate an attack or, if the actor is state-sponsored, create serious international implications. Only approved federal government or military personnel are permitted to counterattack.

### International Standpoint on Counterattacks

The International Court of Justice (ICJ) allows a counterattack only if it: affects only the original attacking party, is a direct communication asking the attacker to stop, does not escalate the situation, and has reversible effects. In practice, these conditions are hard to measure, so organizations generally avoid counterattacking.

### Ethical Principles and Methodologies

- **Confidentiality:** Only authorized users can access specific assets or data; professionally, this means respecting privacy to safeguard private data.
- **Privacy protection:** Safeguarding personal information from unauthorized use. PII can identify a person (name, phone number); SPII is a stricter subset (SSNs, credit card numbers).
- **Laws:** Security professionals have an ethical obligation to remain unbiased, work honestly and within the law, be transparent, rely on evidence, and stay informed to keep advancing their skills.

**Key takeaway:** Because counterattacking is largely disallowed, the field relies on frameworks and controls (like the CIA triad) to address confidentiality, privacy, and legal obligations. Ethics play a large role in daily decision-making when responding to threats or breaches.

## Glossary Terms from Module 3

| Term | Definition |
|------|------------|
| Asset | An item perceived as having value to an organization |
| Attack vectors | The pathways attackers use to penetrate security defenses |
| Authentication | The process of verifying who someone is |
| Authorization | The concept of granting access to specific resources in a system |
| Availability | The idea that data is accessible to those who are authorized to access it |
| Biometrics | The unique physical characteristics used to verify a person's identity |
| Confidentiality | The idea that only authorized users can access specific assets or data |
| CIA triad | A model that helps inform how organizations consider risk when setting up systems and security policies |
| Detect | A NIST core function related to identifying potential security incidents and improving monitoring |
| Encryption | The process of converting data from a readable format to an encoded format |
| Identify | A NIST core function related to management of cybersecurity risk and its effect on people and assets |
| Integrity | The idea that data is correct, authentic, and reliable |
| NIST CSF | A voluntary framework of standards, guidelines, and best practices to manage cybersecurity risk |
| NIST S.P. 800-53 | A unified framework for protecting the security of information systems within the U.S. federal government |
| OWASP | A non-profit organization focused on improving software security |
| Protect | A NIST core function used to implement policies, procedures, training, and tools that mitigate cybersecurity threats |
| Recover | A NIST core function related to returning affected systems back to normal operation |
| Respond | A NIST core function related to containing, neutralizing, and analyzing security incidents |
| Risk | Anything that can impact the confidentiality, integrity, or availability of an asset |
| Security audit | A review of an organization's security controls, policies, and procedures against a set of expectations |
| Security controls | Safeguards designed to reduce specific security risks |
| Security frameworks | Guidelines used for building plans to help mitigate risk and threats to data and privacy |
| Security posture | An organization's ability to manage its defense of critical assets and data and react to change |
| Threat | Any circumstance or event that can negatively impact assets |

## Personal Reflection

This module tied together how frameworks, controls, and compliance aren't separate concerns but work as a system. Frameworks set the goals, controls implement them, and compliance verifies they're being met. The legal side around counterattacks was new territory for me: I hadn't realized how firmly "defend only" is codified into U.S. law, or how narrow and impractical the ICJ's conditions for an international counterattack actually are. It reinforced that ethical, evidence-based, within-the-law decision-making is as core to this job as the technical skills.
